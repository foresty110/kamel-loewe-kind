# 게시판 와이어프레임 + 기획서 페이지

## 개요

각 기획서 문서는 왼쪽 **SCREEN** 패널(가짜 브라우저 창 안에 만든 UI 와이어프레임)과 오른쪽 **SPEC** 패널(번호가 매겨진 각 UI 요소를 설명하는 표)을 짝지어 보여준다. 기준 자료는 `ebrain/jspBoard/ebrainstudy-template-week-1/[eBrainSoft] Study - Board Wireframe V1.1.pdf`이다. `docs/board-list.html`이 이미 승인된 정본(canonical) 예시이며, 새 페이지는 이 디자인 시스템을 그대로 재사용하고 내용만 바꾼다.

## 진행 순서

1. 해당 화면의 필드 목록과 목업 레이아웃을 확인하기 위해 PDF에서 맞는 페이지를 읽는다: 5페이지 = 게시판 보기, 6페이지 = 게시판 등록, 7페이지 = 게시판 수정, 8페이지 = 비밀번호 확인.
2. 이 스킬 폴더에 있는 `template.html`을 `docs/<화면이름>.html`로 복사한다 (예: `board-view.html`). 공유 `<style>` 블록이 이미 다 들어 있으므로 CSS는 새로 쓰지 말고, 표시해 둔 두 개의 placeholder 영역에만 화면별 마크업을 채워 넣는다.
3. `.doc-header`(화면 명 / 화면 아이디 / 화면 경로)와 `.browser-address`의 URL을 PDF에 나온 화면 경로에 맞게 채운다.
4. 사이트의 철학 커뮤니티 톤에 맞는 사실적인 목업 콘텐츠로 SCREEN 와이어프레임 본문을 만든다 (톤 참고는 `docs/board-list.html`: 윤리학/철학사/인식론 같은 카테고리, 한국어 이름, 철학 주제의 게시글 제목).
5. SCREEN 목업에서 쓴 배지 번호와 1:1로 대응하는 SPEC 표 행을 만든다. PDF의 화면 설명 불릿을 그대로 옮기거나 필요한 만큼만 다듬어 짧게 유지한다.
6. 다 만들었다고 하기 전에 검증한다 — 아래 "검증" 항목 참고.

## 가장 중요한 규칙: 주석 배지는 UI를 절대 밀어내지 않는다

SCREEN 목업의 모든 번호 배지는 요소 위에 떠 있어야 하며, 요소 앞에 인라인으로 끼어들면 안 된다:

```html
<!-- 틀린 예: 배지가 인라인으로 들어가서 "조회수"가 원래 위치보다 오른쪽으로 밀림 -->
<th><span class="badge">6</span>조회수</th>

<!-- 맞는 예: 배지가 position:absolute로 위-왼쪽에 떠 있고, "조회수"는 실제 화면에서 렌더링될 자리 그대로 있음 -->
<th class="anno"><span class="badge-pin">6</span>조회수</th>
```

`.badge-pin`은 `position:absolute`이며 `.anno`(`position:relative`) 래퍼 안에 있어야만 제대로 위치를 잡는다. `.badge`(위치 지정 없는 일반 인라인 원)는 SPEC 표 자체의 번호 열에서만 쓰고, SCREEN 목업 안에서는 절대 쓰지 않는다.

PDF 목업 이미지가 같은 배지 번호를 서로 다른 두 지점에 중복해서 쓰는 경우가 있다(원본 PDF가 완벽히 일관적이지 않음). 이럴 때는 실제로 몇 개의 항목/번호가 존재하는지에 대해 PDF의 화면 설명 표를 기준으로 삼는다. 그 행의 불릿 내용이 실제로 설명하는 지점에만 배지를 달고, 나머지 지점은 배지 없이 두거나(또는 이웃한 행의 설명으로 자연스럽게 커버되게 하고) 없는 행을 새로 지어내지 않는다.

## 디자인 토큰 (재발명하지 말고 template.html에서 그대로 가져올 것)

- 색상: `--accent:#4f46e5` 인디고, `--screen-bg:#eef1fd`, `--spec-bg:#f7f7f9`, `--ink`/`--muted`/`--line` 중성색.
- 레이아웃: `.split`은 flex row이며 `.column.screen{flex:2.2}` vs `.column.spec{flex:1}` — SCREEN이 눈에 띄게 더 넓어야 한다. `.divider`는 "화면 ↔ 기획서" 라벨이 세로로 돌아간 점선 세로줄이다.
- SCREEN 크롬(chrome): `.browser-window > .browser-titlebar`(신호등 점 + 가운데 정렬된 주소창)가 `.wireframe-panel`을 감싼다.

## 자주 하는 실수 (board-list.html을 만들며 실제로 겪은 것들)

| 실수 | 해결 |
|---|---|
| 배지를 라벨 텍스트 앞에 인라인으로 배치 | 요소를 `.anno`로 감싸고 `.badge-pin`(absolute)을 쓴다 — `.badge`가 아니다 |
| 배지에 말풍선 "꼬리" 추가 | 하지 말 것 — 이 크기에서는 삐뚤빼뚤하고 깨진 모양으로 렌더링된다. 테두리만 있는 원이 떠 있는 것만으로도 주석처럼 충분히 읽힌다 |
| 긴 한글 SPEC 텍스트가 단어 중간에서 잘려 보임 | 템플릿에 전역으로 넣어 둔 `word-break: keep-all`이 이미 처리한다 — 지우지 말 것 |
| 페이지가 마지막 SPEC 행까지 스크롤되지 않음 | `.split`을 `min-height: calc(100vh - Npx)`로 고정하지 말 것 — 템플릿처럼 콘텐츠 높이를 그대로 따라가게 둔다 |
| SCREEN 컬럼이 SPEC 옆에서 좁아 보임 | 2.2 : 1 flex 비율을 유지(또는 더 넓게)한다 — SPEC은 설명 표 하나만 담으면 되므로 좁아도 된다 |
| PDF 목업 이미지 안의 UI 요소를 빠뜨림 (예: 목록 화면의 등록일 범위 조회 필드 — 번호가 따로 없어서 화면 설명 표의 불릿 문장 안에만 언급되고 별도 배지가 없었음) | 번호가 매겨진 항목만 보고 만들지 말고, PDF 목업 이미지 자체를 다시 살펴서 눈에 보이는 UI 요소를 전부 옮겼는지 확인한다. 배지 번호가 없는 요소도 화면에는 존재할 수 있다 |

## 검증

`docs/` 디렉터리에서 로컬로 서버를 띄우고(`python3 -m http.server`) 실제 브라우저(Playwright MCP 도구)로 대략 1600×900 크기에서 새 페이지를 열어 확인한다. 텍스트가 잘리지 않는지, 배지-핀이 주변 텍스트와 겹치지 않는지, 맨 아래까지 스크롤했을 때 마지막 SPEC 행이 완전히 보이는지, SCREEN 목업의 필드 목록이 원본 PDF 페이지와 일치하는지 확인한 뒤에야 완료로 본다.
